Latihan: Membangun SPA Pertama

Sebelum memahami implementasi View Transition API pada single-page application (SPA), penting untuk memahami cara dasar membangun SPA. Ini memungkinkan pengguna berpindah antar tampilan tanpa memuat ulang halaman, menciptakan pengalaman navigasi yang mulus dan cepat.

Persiapan

Ada dua pendekatan utama dalam membangun SPA: mode history dan mode hash. Mode history memiliki keuntungan yang lebih daripada mode lain, seperti mempertahankan kemampuan navigasi browser dan SEO. Namun, mode history cukup menantang untuk diimplementasikan karena perlu banyak manipulasi history browser melalui History API. SPA mode history, biasanya hadir sekaligus dengan penerapan framework seperti Angular, Vue.js, atau menggunakan library seperti react-router.

Pada kelas ini, kami tidak menerapkan framework spesifik, tidak pula membahas detail tentang routing. Kami menggunakan mode hash karena implementasinya sederhana, tetapi esensi penerapan transisi dan animasi pada SPA tetap Anda dapatkan.

Di akhir latihan ini, kita akan membangun aplikasi sederhana seperti berikut.

Mari kita mulai pengembangannya!


Latihan

Dalam mempercepat langkah latihan, kita akan menggunakan starter project untuk memulai pengembangan SPA. Silakan unduh dan buka spa-pertama-starter-project.zip menggunakan code editor kesayangan Anda.

Pertama-tama, silakan jalankan proyeknya dan seharusnya akan terlihat seperti berikut.

Hanya ada sebuah halaman sederhana dengan satu buah elemen <a>. Sip!

Mungkin Anda sudah mencoba klik elemen link di atas, ya? Tidak ada perubahan view di sana. Namun, kita melihat ada perubahan URL pada address bar dan log baru dalam console browser.

Perubahan URL yang terjadi akan kita gunakan untuk melakukan navigasi halaman. Di sini, kita perlu membangun routing system sehingga pengguna bisa menuju ke halaman about (about page). Routing system tersebut akan menangani pergantian atau perubahan jalur (route) yang menentukan halaman atau konten yang ditampilkan kepada pengguna berdasarkan URL. Perubahan jalur ini akan kita sebut perubahan routing. Lalu, bagaimana tekniknya?

Kita bisa melihat kode bercetak tebal dalam index.js berikut.

index.js
document.addEventListener('DOMContentLoaded', () => {
  console.log('Hello, world!');
});

window.addEventListener('hashchange', () => {
  console.log('Hash changed!');
});

Event hashchange akan dibangkitkan (triggered) ketika ada perubahan hash pada URL. Kita akan memanfaatkan event ini untuk melacak perubahan routing. Oleh karena itu, kita akan merencanakan beberapa route berikut.

  • “/” ⇒ halaman utama (home page).
  • “/about” ⇒ halaman about (about page).

Kami harap, sekarang Anda sudah mulai memiliki gambarannya, ya. Selanjutnya, kita akan mulai menuliskan banyak kode. Berikut daftar urutan tugas kita berikutnya.

  1. Menentukan konten setiap halaman.
  2. Membangun routing system.
  3. Melakukan perubahan konten berdasarkan perubahan routing.

Kita mulai dari urutan pertama.

Silakan buat berkas baru bernama “routes.js” dalam direktori “routes” dan isikan dengan kode berikut.

class HomePage {
  async render() {
    return `
      <h1 class="content-title">Home Page</h1>
      <p>Ini adalah konten halaman utama.</p>
      <p>Mari kita cek <a href="#/about">halaman about</a>!</p>
    `;
  }

  async afterRender() {
    // Do something...
  }
}

class AboutPage {
  async render() {
    return `
      <h1 class="content-title">About Page</h1>
      <p>Ini adalah konten halaman about.</p>
      <ol>
        <li>Kita</li>
        <li>punya</li>
        <li>elemen</li>
        <li>daftar</li>
        <li>di sini!</li>
      </ol>
      <p>Oke, sudah cukup. Kita bisa kembali ke <a href="#/">halaman home</a>.</p>
    `;
  }

  async afterRender() {
    // Do something...
  }
}

const routes = {
  '/': () => new HomePage(),
  '/about': () => new AboutPage(),
};

export default routes;

Kita memiliki variabel bernama routes dalam bentuk objek untuk mendefinisikan daftar nama jalur (route name) yang tersedia pada website. Masing-masing route diarahkan pada objek berbeda berdasarkan halaman yang ingin ditampilkan.

Karena membutuhkan dua halaman, kita buat dua buah objek yang mewakili masing-masing halaman. Ada dua method yang kita lihat: render dan afterRender. Method render berguna untuk mendefinisikan struktur konten halaman dan akan di-render ke DOM, sedangkan method afterRender berguna untuk menjalankan tugas-tugas tertentu setelah DOM diperbarui. Keduanya wajib kita tambahkan pada setiap objek halaman.

Keren! Tugas pertama sudah selesai.

  1. Menentukan konten setiap halaman.
  2. Membangun routing system.
  3. Melakukan perubahan konten berdasarkan perubahan routing.

Konten halaman sudah disiapkan, lalu, bagaimana cara kita menangani perubahan routing berdasarkan URL?

Browser memiliki API untuk mengakses URL yang sedang aktif, terutama hash. Kita bisa mengaksesnya melalui window.location.hash. Kita bisa memanfaatkan properti tersebut untuk mendapatkan nilai URL hash. Misalnya berikut.

  • “http://127.0.0.1:5500/#/” ⇒ #/
  • “http://127.0.0.1:5500/#/about” ⇒ #/about
  • “http://127.0.0.1:5500/#/products/laptop” ⇒ #/products/laptop

Namun, tentunya kita tidak membutuhkan karakter hash “#” dalam routing. Cukup ambil nama jalurnya saja atau kita sebut pathname. Oleh karena itu, silakan buat berkas “url-parser.js” dan isikan dengan kode berikut.

url-parser.js
function extractPathnameSegments(pathname) {
  const splitPath = pathname.split('/');

  return {
    resource: splitPath[1] || null,
    id: splitPath[2] || null,
  };
}

function constructRouteFromSegments(pathSegments) {
  let pathname = '';

  if (pathSegments.resource) {
    pathname = pathname.concat(`/${pathSegments.resource}`);
  }

  if (pathSegments.id) {
    pathname = pathname.concat('/:id');
  }

  return pathname || '/';
}

function getActivePathname() {
  return location.hash.replace('#', '') || '/';
}

export function getActiveRoute() {
  const pathname = getActivePathname();
  const pathSegments = extractPathnameSegments(pathname);

  return constructRouteFromSegments(pathSegments);
}

Sebetulnya, untuk menentukan halaman yang akan ditampilkan, kita perlu mendapatkan nama route secara tepat berdasarkan URL yang diminta. Namun, kita perlu melakukan beberapa tahapan–bungkus dalam empat function–sebelum itu terjadi. Berikut penjelasannya.

  • getActivePathname
    Function ini berfokus pada mendapatkan nilai URL hash yang sedang aktif. Untuk melakukannya, kita membutuhkan location dalam mendapatkan URL terbaru. 
  • extractPathnameSegments
    Function ini bekerja untuk mengekstraksi masing-masing segmen dalam pathname. Path pertama kita sebut sebagai resource. Misalnya, jika resource bernilai “about”, pengguna sedang meminta halaman yang berkaitan dengan halaman about.

    Selain itu, method ini juga bermanfaat dalam mendapatkan data dinamis. Ini terlihat dengan adanya properti id dalam nilai kembaliannya. Misalnya: “http://127.0.0.1:80/#/products/laptop”, “http://127.0.0.1:80/#/products/smartphones”, dsb. Penggunaannya akan kita manfaatkan pada praktik berikutnya. 
  • constructRouteFromSegments
    Setelah masing-masing segmen dari pathname dilebur, kita perlu menentukan nama route yang diminta pengguna menggunakan function ini. Dengan demikian, kita bisa mengarahkan pengguna kepada konten yang dicari.
  • getActiveRoute
    Setiap function yang terlibat dalam pengolahan pathname sebelumnya akan membentuk nama route. Function ini akan menjalankan tahapan-tahapan sebelumnya dan digunakan pada bagian aplikasi yang membutuhkannya.

Berikutnya, mari kita implementasikan function getActiveRoute pada berkas index.js. 

import routes from './routes/routes.js';
import { getActiveRoute } from './routes/url-parser.js';

document.addEventListener('DOMContentLoaded', async () => {
  const routeName = getActiveRoute();
  const route = routes[routeName];

  console.log('routeName:', routeName);
  console.log('route:', route);
});

window.addEventListener('hashchange', async () => {
  const routeName = getActiveRoute();
  const route = routes[routeName];

  console.log('routeName:', routeName);
  console.log('route:', route);
});

Jika sudah, silakan Anda cek hasilnya. Jangan lupa cek bagian console browser, ya!

Keren! Tidak salah lagi, perubahan routing sudah bisa kita atasi dan tersisa satu tugas lagi.

  1. Menentukan konten setiap halaman.
  2. Membangun routing system.
  3. Melakukan perubahan konten berdasarkan perubahan routing.

Kebanyakan website biasanya hanya melakukan perubahan pada konten utamanya saja. Biasanya, konten ini ditampung dalam <main>.

Perhatikan kode bercetak tebal berikut.

index.html
<body>
  <header class="main-header">
    <div>SPA Pertama</div>
  </header>

  <main id="content" class="content">
    <h1 class="content-title">Home Page</h1>
    <p>Ini adalah konten halaman utama.</p>
    <p>Mari kita cek <a href="#/about">halaman about</a>!</p>
  </main>
</body>

Berbeda dengan <main>, memang elemen <header> adalah konten yang jarang berubah. Oleh karena itu, kita hanya perlu mengatur konten dalam <main> agar mampu menampung konten secara dinamis berdasarkan perubahan routing.

Mari kita tangkap saja elemen ini menggunakan ID dalam index.js.

index.js
import routes from './routes/routes.js';
import { getActiveRoute } from './routes/url-parser.js';

const content = document.querySelector('#content');

document.addEventListener('DOMContentLoaded', async () => {
  /* ... */
});

window.addEventListener('hashchange', async () => {
  /* ... */
});

Dari sini, seharusnya Anda paham hal yang harus dilakukan, bukan? Cukup simpel, kita manfaatkan saja innerHTML untuk mengubah konten <main>. Silakan sesuaikan kodenya seperti berikut.  

index.js
import routes from './routes/routes.js';
import { getActiveRoute } from './routes/url-parser.js';

const content = document.querySelector('#content');

document.addEventListener('DOMContentLoaded', async () => {
  const routeName = getActiveRoute();
  const route = routes[routeName];

  const page = route();
  content.innerHTML = await page.render();
});

window.addEventListener('hashchange', async () => {
  const routeName = getActiveRoute();
  const route = routes[routeName];

  const page = route();
  content.innerHTML = await page.render();
});

Ahay! Sekarang, kita tinggal coba jalankan dan nikmati hasilnya.

Perlu dicatat dengan saksama bahwa dalam routing system menggunakan mode hash, kita wajib tambahkan karakter hash “#” sebelum pathname saat membuat elemen navigasi. Hal ini karena perubahan route akan ditentukan dari perubahan nilai hash. Misalnya perubahan dari “#/” menjadi “#/about” akan menentukan perubahan route.

Selain itu, perubahan route menggunakan hash juga tidak akan memicu browser untuk reload. Ini juga menjadi salah satu indikasi aplikasi mengadopsi arsitektur SPA.

Catatan:
Teks setelah hash “#” sebetulnya memiliki nama teknisnya: fragment identifier. Umumnya, ia digunakan untuk mengarah ke suatu bagian atau tampilan dari halaman web. Karena tidak menimbulkan reload dan masuk ke browser history list, kita mampu memanfaatkannya untuk menerapkan SPA sederhana.

Sebelum lanjut, kita kosongkan dahulu konten dalam <main> yang terukir pada index.html, ya.

<body>
  <header class="main-header">
    <span>SPA Pertama</span>
  </header>

  <main id="content" class="content">
    <!-- Dynamic content will be inserted here -->
  </main>
</body>

 

Membersihkan Kode

Kita melihat ada duplikasi kode, bukan? Ini terjadi dalam dua event handler dalam index.js. Bukanlah hal yang fatal–buktinya website berjalan normal, tetapi ini menimbulkan redundansi kode dan sulit dipelihara. Oleh karenanya, kita akan membuat satu buah class bernama “App”.

Silakan buat berkas app.js dalam direktori scripts dan isikan dengan kode berikut.

import { getActiveRoute } from './routes/url-parser.js';
import routes from './routes/routes.js';

export default class App {
  #content;

  constructor({ content }) {
    this.#content = content;
  }

  async renderPage() {
    const routeName = getActiveRoute();
    const route = routes[routeName];

    // Get page instance
    const page = route();

    this.#content.innerHTML = await page.render();
  }
}

Class di atas berguna sebagai pusat pengelolaan aplikasi di sisi JavaScript. Dalam arti lain, ia akan menjadi root application.

Nice to know:
Dalam beberapa UI framework/library yang mengadopsi component architecture sebagai konsep pembangunan user interface, mereka memiliki satu komponen induk dari berbagai komponen di bawahnya. Perilaku ini mirip dengan class App yang sedang kita bangun.

Jika sudah, kita perlu memperbarui kode dalam berkas index.js. Silakan sesuaikan dengan kode berikut.

index.js
import App from './app.js';

document.addEventListener('DOMContentLoaded', async () => {
  const content = document.querySelector('#content');
  const app = new App({ content });
  await app.renderPage();

  window.addEventListener('hashchange', async () => {
    await app.renderPage();
  });
});

Silakan simpan semua perubahan kode serta pastikan tidak ada kode yang menimbulkan error dan bug.

Selamat! Anda sudah berhasil membangun satu buah aplikasi berarsitektur SPA dengan baik. Berikutnya, kita masih akan bermain dengan arsitektur ini, tetapi dengan menerapkan pola Model-View-Presenter. Waw, apa itu? Ayo, tunggu apa lagi, lanjut!
